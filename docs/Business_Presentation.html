Business Problem

The Client's Question

"How much revenue is the client getting from external search engines, such as Google, Yahoo and MSN, and which keywords are performing the best based on revenue?"

Why This Matters

Marketing ROI

The client invests in SEO and SEM to drive traffic from search engines. They need to know which keywords generate revenue — not just traffic — to optimize their marketing spend.

Budget Allocation

Without keyword-level revenue data, the client is spending marketing budget blindly. This analysis enables data-driven decisions on where to invest.

Competitive Advantage

Understanding which search engines convert best allows the client to outperform competitors by focusing on high-ROI channels.

Keyword Strategy

Some keywords drive traffic but no revenue (e.g., "cd player"). Identifying these saves wasted ad spend and redirects budget to profitable keywords.

The Challenge

Why Simple Referrer Lookup Doesn't Work

A visitor's journey spans multiple page hits. The search engine referrer only appears on the first hit. By the time they purchase, the referrer is an internal checkout page.

The Problem

If we only look at the referrer on the purchase row, we'd attribute $0 to Google — the referrer there is an internal checkout page. We need session-based attribution to trace the revenue back to the original search engine keyword.

Our Approach

Session-Based Revenue Attribution

We track each visitor's session by IP address, recording the search engine and keyword on their first hit, then attributing revenue when a purchase event occurs.

Step 1 — Detect Search Engine

Parse the referrer URL on each hit. If it matches Google, Bing, Yahoo, or MSN, extract the search keyword:

Referrer URLEngineKeyword
google.com/search?q=IpodGoogleIpod
bing.com/search?q=ZuneBingZune
yahoo.com/search?p=cd+playerYahoocd player

Step 2 — Track Session

Store the first search engine referral per visitor IP. Only the first referral is kept (first-touch attribution):

Visitor IPEngineKeyword
67.98.123.1google.comIpod
23.8.61.21bing.comZune
44.12.96.2google.comipod
112.33.98.231yahoo.comcd player

Step 3 — Attribute Revenue on Purchase

When event_list contains "1" (Purchase), look up the visitor's session and extract revenue from product_list. The revenue is attributed to the stored (engine, keyword) pair.

Business Insights

Key Findings & Recommendations

1. Google is the Dominant Revenue Driver — $480 (65.8%)

Both Google keywords ("Ipod" and "ipod") converted to purchases. Google drives the majority of search revenue.

Recommendation: Continue investing in Google SEO/SEM for electronics keywords. Consider expanding to related terms like "mp3 player", "portable music".

2. Bing Delivers High-Intent Buyers

Bing generated $250 from a single visitor with the fastest conversion path (5 hits, 21 minutes). The visitor went directly: search → product → cart → purchase.

Recommendation: Don't overlook Bing. Despite lower traffic volume, Bing visitors show higher purchase intent. Consider increasing Bing ad spend for high-margin products.

3. Yahoo Has Traffic but Zero Revenue

The Yahoo visitor searched "cd player" but left after one page. This indicates a keyword-product mismatch — the visitor's intent didn't match the site's offerings.

Recommendation: Either stock CD players and create relevant landing pages, or exclude "cd player" from Yahoo ad targeting to avoid wasted impressions.

4. Product Comparison Drives Higher Revenue

Visitor 1 compared Ipod Nano ($190) and Ipod Touch ($290), then bought the premium option. Visitor 3 went straight to Nano and bought it. Comparison shoppers spent 53% more.

Recommendation: Feature product comparison tools prominently on product pages. Visitors who compare tend to buy premium options.

Technical Decisions

Design Decisions & Trade-offs

Decision Why Trade-off
IP-based sessions Only consistent visitor identifier in the data. No cookies or visitor IDs available. Multiple users behind same corporate IP would merge. Use IP + User Agent in production.
First-touch attribution Credits the channel that originally discovered the customer. Standard for SEO/SEM analysis. Last-touch would credit the most recent channel. Both are valid models.
Config-driven architecture Adding a new search engine = one line in YAML. No code changes needed. Different clients can use different configs. Slight overhead of YAML parsing. Negligible for this use case.
Streaming I/O File is read line-by-line, never fully loaded into memory. Handles files larger than available RAM. Cannot sort the input. But we only sort the output (small result set).
Case-sensitive keywords "Ipod" and "ipod" are kept as separate keywords to preserve original search queries. Could normalize to lowercase for combined reporting. Configurable per client preference.
AWS Lambda (serverless) No servers to manage. Pay only when processing. Auto-scales. Free tier covers this workload. 15-minute timeout limits file size to ~5GB per invocation. Larger files need EMR/Glue.

Scalability

Scaling to 10GB+ Files

The current application streams files line-by-line (memory efficient). The bottleneck for large files is the session tracking dictionary that grows with unique visitor IPs.

File Size Unique IPs Memory Needed Recommended Approach
< 1 GB ~100K ~10 MB Current approach ✓
1 – 10 GB ~1M ~100 MB Current approach ✓
10 – 50 GB ~5M ~500 MB SQLite-backed session store
50 – 500 GB ~50M ~5 GB AWS Glue + PySpark
500+ GB ~100M+ ~10+ GB Amazon EMR + Spark cluster

Immediate Next Step for 10GB+ Files

Replace the in-memory Python dictionary with SQLite — a disk-backed database built into Python. This requires minimal code changes and handles billions of session records efficiently.
